Client-Side vs Server-Side Logging (Futuristic SN Theme)

Use client logging for browser-side debugging (client console), use server logging for platform-side tracing (server syslog), and use UI messages for end-user feedback (UI form).

Client: DevTools Console Server: System Logs (syslog) UI: Form messages Rule: don’t mix contexts

Core Comparison Where it runs • where it prints

Category Client-side Logging Server-side Logging
Runs on Browser (user session) ServiceNow server
Typical scripts Client Script, Catalog Client Script, UI Page client code, Portal client controller Business Rule, Script Include, Scheduled Script, Fix Script, Flow/Workflow server actions
Purpose UI + form behavior debugging Backend + record processing tracing
Primary APIs jslog(), console.log(), console.warn(), console.error() gs.info(), gs.warn(), gs.error(), gs.debug(), gs.log()
Where to view DevTools → Console System Logs → All (syslog.list)
Database access ❌ No ✅ Yes (GlideRecord)
Who sees it Developer in that browser session Admins/devs with log access
Do not confuse with addInfoMessage/addErrorMessage (UI banners) — user feedback, not system logs.
⚠️
Context rule: gs.* is server-only. console.* / jslog() is client-only. Using the wrong API in the wrong layer will not behave as expected.

Client Methods console output

Method Use
jslog(message) ServiceNow helper — prints to browser console (SN-tagged).
console.log(message) General debugging output.
console.warn(message) Warnings for non-blocking issues.
console.error(message) Errors for failures/exceptions.
// Client Script (browser) function onChange(control, oldValue, newValue, isLoading) { jslog("onChange fired: newValue=" + newValue); console.log("caller_id=" + g_form.getValue("caller_id")); if (!newValue) console.warn("Empty value detected"); }
View: F12 → Console. Client logs are developer-only and scoped to the current browser session.

Server Methods (GlideSystem) syslog output

Method Level Use
gs.info(message) INFO Operational message.
gs.warn(message) WARN Unexpected but non-fatal condition.
gs.error(message) ERROR Failure / critical condition.
gs.debug(message) DEBUG Debug traces (depends on debug visibility).
gs.log(message, source) CUSTOM Custom source label for filtering/tracing.
// Business Rule (server) gs.info("BR started: " + current.getTableName() + " " + current.sys_id); gs.warn("Priority=" + current.priority); gs.error("Missing assignment_group for " + current.number); gs.log("Trace checkpoint", "INCIDENT_RULE");
View: System Logs → All (syslog.list). Server logs are instance-wide.

UI Messages (Not Logging) form output

Goal Client API (Form UI) Server API (Form UI)
Info banner g_form.addInfoMessage("...") gs.addInfoMessage("...")
Error banner g_form.addErrorMessage("...") gs.addErrorMessage("...")
Field message g_form.showFieldMsg("field","msg","error|info") Typically done on client; server-side requires UI patterns (not a log)
// Client UI messages (end-user feedback) function onLoad() { g_form.addInfoMessage("Verify mandatory fields before saving."); } function onSubmit() { if (!g_form.getValue("short_description")) { g_form.addErrorMessage("Short description is mandatory."); g_form.showFieldMsg("short_description", "Required", "error"); return false; } }
UI messages are visible to users and should not be used as developer logging in production.

Quick Decision Guide choose the correct channel

Need Use Output
Confirm client script executed / inspect values jslog() / console.log() Console
Trace backend logic / database updates gs.info() / gs.warn() / gs.error() Syslog
Notify user / enforce UX validation g_form.add*Message() / showFieldMsg() Form UI
